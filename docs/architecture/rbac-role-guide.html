Platform Design · RBAC Best Practices

Role Hierarchy &
Access Control Guide

Who sees what, who can do what, and why — across every level of the Sovereign Knowledge Platform. Based on industry standards from Confluence, Notion, and leading RAG platforms.

The guiding principle

Every enterprise knowledge platform that scales successfully follows one rule: users should only ever see the level of the system that they own and operate. A workspace member should never see organization billing. An org admin should never see other organizations. This creates trust, reduces cognitive overload, and prevents accidental data exposure — which in a RAG system is a critical security boundary, not just a UX preference.

The hierarchy you're building mirrors exactly what Confluence, Notion Enterprise, and Guru use — a three-tier model: Platform → Organization → Workspace. Each tier has its own admin, and admins at each tier cannot escalate their access upward.

🔒

Security principle (from OWASP and Pinecone): In RAG systems, access control must be enforced at retrieval time — not just in the UI. Even if a user navigates to a URL they shouldn't access, the backend query must filter to only documents their role permits. UI hiding is the first layer; query-time filtering is the security layer.

The 5-role hierarchy

👑 Platform Owner
God mode · All orgs · All settings
🏛 Org Admin
One organization · All its workspaces · No platform
🗂 Workspace Admin
Assigned workspaces only · No org or platform
📝 Workspace Editor / Contributor
Can upload docs, manage connectors in their workspace
💬 Member (Read-Only / Chat User)
Can only ask questions and read answers · No management

Critical security and UX rules

🔒

Never show locked nav items — hide them entirely. Showing a grayed-out "Billing" link to a Workspace Admin is confusing and feels like a broken product. Hide sections the user has no access to. Only reveal what they can interact with. Notion, Linear, and Slack all follow this pattern.

🛡️

Enforce permissions at the API layer, not just the UI. A determined user can call API endpoints directly if you only hide links in the UI. Every API route must verify role and scope before returning data. This is especially critical in RAG — a query that returns documents from outside the user's workspace is a data breach, not a UX bug.

⚠️

Workspace Admin cannot create workspaces — only Org Admin and above. This is your hierarchy rule from the brief. Workspace Admin manages what they're given. Only Org Admin can create/delete workspaces. This prevents workspace sprawl and keeps the org structure under control.

⚠️

Only Platform Owner can delete an organization. Org Admins can self-serve everything except deletion. This is a standard SaaS safety pattern — irreversible destructive actions require the highest privilege level. Org Admins can archive/suspend, but not hard-delete.

💡

Members should land directly in chat with zero admin chrome. When a Member logs in, route them straight to the chat interface. No sidebar full of locked items. No "you don't have access" pages. Just a clean chat. This is the entire product from their perspective — protect that experience.

💡

Queries must filter to the user's accessible document set. A workspace member should never get an answer sourced from documents in another workspace. Enforce this at the vector search query level by passing the user's workspace_id and user_id as metadata filters — not just by hiding UI elements.

💡

Audit log every admin action, not just queries. Log: who created/deleted what, role changes, connector connects/disconnects, document uploads/deletions, and any impersonation by Platform Owner. Org Admins in compliance-heavy industries (GovCon, legal) will ask for this log during sales calls.

Implementation checklist for your dev team

01

Define roles in your database

Create a roles enum: platform_owner | org_admin | workspace_admin | editor | member. Store user roles in an org_members table with {user_id, org_id, role} and a workspace_members table with {user_id, workspace_id, role}. A user can have different roles in different workspaces.

02

Build a permission middleware for every API route

Every API endpoint needs a guard: requireRole('org_admin', orgId) or requireWorkspaceAccess(workspaceId). The guard reads the user's JWT, fetches their role for the requested resource scope, and rejects with 403 if insufficient. Never trust the frontend to enforce access.

03

Scope all vector search queries by workspace

When a user queries, pass their workspace_id as a metadata filter to pgvector/Pinecone. The query should be: WHERE workspace_id = ? AND (user's accessible doc IDs). Never return documents from outside the user's scope, even if they're semantically relevant.

04

Implement navigation guards in the frontend

On login, fetch the user's effectiveRole and accessibleWorkspaces. Use this to render only the nav items the user can access. Never render admin nav items with a disabled/locked state — simply don't render them. React Router's PrivateRoute or Next.js middleware handle this cleanly.

05

Default landing route by role

Platform Owner → /platform/overview. Org Admin → /dashboard. Workspace Admin → /workspaces/{first_ws}/chat. Editor → /workspaces/{ws}/chat. Member → /chat (fullscreen, no sidebar chrome). Set these in your auth callback after login.

06

Add audit logging middleware

Log every state-changing action to an audit_events table: {event_type, actor_id, actor_role, target_type, target_id, org_id, workspace_id, metadata, timestamp}. This enables the Audit Log page for Org Admins and Workspace Admins and is a strong selling point for compliance-focused buyers.

07

Test each role in isolation

Create a test account for each role tier. Verify: (a) the correct nav renders, (b) API calls from unauthorized roles return 403, (c) vector queries never return cross-workspace documents, (d) the Platform Owner impersonation mode is logged in the audit trail. Run this as a regression test on every deployment.
