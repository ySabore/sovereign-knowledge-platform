For non-technical stakeholders

How the AI Knowledge Assistant works

A plain-English walkthrough of every layer in the system — what it does, why it exists, and what it means for the business. No engineering background required.

The big picture — in one analogy

Think of this product as a brilliant new employee who has read every document your company has ever written — every policy, every past project, every help article, every Slack thread — and can answer any question about any of it in seconds, while pointing to the exact page they got the answer from. They never forget anything, never leave the company, and are available 24 hours a day. The "system" is simply the infrastructure that makes that possible at scale.

What happens when someone asks a question

User types a question
In the app, Slack, or via API
→
Identity verified
Are they allowed to access this workspace?
→
Relevant passages found
AI searches all connected documents
→
Answer generated
Claude reads the passages and writes a response
→
Answer + sources returned
With citations linking back to original docs

This entire flow takes under 3 seconds. The system is built in six layers, each handling a different part of this process.

🖥️
Layer 1 — Client Layer

The surfaces people actually use

Everything a user sees and touches — the windows into the system
What this layer does

This is the front door to the product — every screen, every button, every interface that a real person interacts with. The client layer doesn't do any "thinking" — it collects what the user asks, sends it to the system, and displays what comes back. Think of it as the face of the product.

💬Web App
The main product — a browser-based chat interface where employees type questions and receive cited answers. Works like a specialized version of ChatGPT but only knows your company's documents.
Real-world analogy: Like the Google search bar, but instead of searching the whole internet, it searches only your company's knowledge — and explains its sources.
📱Mobile App
The same experience on a phone. Useful for field teams, sales reps on the go, or anyone who needs to look something up away from their desk.
💼Slack Bot
Employees can ask questions directly inside Slack — the tool they already use all day — without switching to another app. Type /ask and get a cited answer in the same thread. This is the biggest driver of daily usage.
Why it matters: Products people use in context win. If you have to leave Slack to find an answer, most people won't bother.
🔌API Consumers
For businesses that want to embed the answer engine into their own internal tools, customer portals, or automated workflows. A company could put "Ask our knowledge base" into their own website.
📊Admin Dashboard
A control panel for team leaders. Shows: how many questions were asked, which questions couldn't be answered (gaps in the knowledge base), how many documents are indexed, and team usage stats.
🔗Connector UI
Where admins connect the product to the company's tools — Confluence, Google Drive, Notion, GitHub. A simple "Connect" button that links the account and starts syncing documents automatically.
"Sarah, a new consultant at a GovCon firm, needs to know the SLA terms in the company's vendor contracts. Instead of pinging the legal team, she types the question into the web app. In 2 seconds she gets the answer with a link to page 12 of the Vendor Policy document."

Layer 2 — API Gateway & Auth

The security checkpoint

Every request passes through here — identity verified, permissions checked, traffic controlled
What this layer does

Before any question reaches the AI engine, it passes through a security checkpoint. This layer answers three questions: Who are you? (authentication), What are you allowed to see? (permissions), and Are you using the system responsibly? (rate limiting). If any check fails, the request stops here.

🪪Auth Service (Clerk)
Manages who can log in and how. Supports sign-in with Google or GitHub, company SSO (single sign-on), and multi-factor authentication. Handles the entire login experience so the team doesn't have to build it.
Analogy: The front desk receptionist who checks your badge before letting you into the building. Knows who you are and which floors you're allowed on.
🛡️RBAC Engine
Role-Based Access Control — determines which documents each user is allowed to see answers from. A sales rep asking about engineering architecture should get "I can't find that" because they don't have access to engineering docs.
Why it matters for enterprise: Without this, a junior employee could potentially ask the AI about confidential HR or legal documents. RBAC prevents that entirely.
📋Audit Logger
Records every single action in the system — who asked what, when, which documents were accessed, and what answer was given. Cannot be edited or deleted. Required for compliance audits and security reviews.
Compliance value: In regulated industries (finance, healthcare, government), auditors need to see a complete trail of information access. This makes that possible.
🚦Rate Limiter
Prevents any single user or organization from overwhelming the system — like a bouncer managing a queue. Also enforces plan limits (e.g., the Starter plan allows 500 queries per month). Returns a clear error when limits are reached.
Key business question this answers

"How do we make sure sensitive company information doesn't leak to people who shouldn't see it?" — This layer is the complete answer to that question. Every query is permission-checked before the AI ever sees it.

⚙️
Layer 3 — Application Services

The business logic engine

The team of specialists that handle every different type of task — orchestrating the work behind the scenes
What this layer does

Once a request passes security, the application services take over. Think of this as a team of back-office specialists — one handles incoming questions, one manages document processing, one keeps connectors in sync, one handles billing. Each service does one job and does it well.

❓Query + Answer Service
The conductor. Takes the user's question, tells the Retriever to find relevant passages, sends everything to the AI engine, receives the answer, and formats it with citations before returning it to the user. This is the central orchestrator.
📥Ingestion Service
When a new document is added (either via connector sync or file upload), this service processes it: cleans the text, cuts it into searchable pieces, converts each piece into a mathematical representation (embedding), and stores it for searching.
Analogy: Like a librarian who receives a new book, reads it, writes index cards for every chapter, and files them in the card catalogue — except they do this for thousands of documents automatically.
🔄Sync Orchestrator
Automatically keeps connected data sources up to date. Every 4 hours, it checks each connected tool (Confluence, Drive, etc.) for new or changed documents and re-indexes anything that's changed. You connect once and it stays current forever.
💳Billing Service
Manages subscriptions, plan limits, and payment processing via Stripe. When a team upgrades from Team to Business, this service updates their limits instantly. Handles failed payments with grace periods before restricting access.
📣Notification Service
Sends email alerts and Slack notifications — for example, when a connector sync fails, when a team is approaching their query limit, or when the admin weekly digest is ready.
📈Analytics Service
Tracks product usage in real time — queries per day, most active users, top questions, unanswered questions, and connector utilization. Feeds the Admin Dashboard. Also identifies gaps in the knowledge base (frequent questions with low confidence answers).
"A law firm connects their Google Drive on Monday. By Monday evening, the Sync Orchestrator has processed 847 documents. On Wednesday, a paralegal updates a client memo in Drive. By Wednesday evening, the system has automatically re-indexed the new version. The paralegal asking about it Thursday morning gets the updated answer."
🧠
Layer 4 — AI / RAG Pipeline

The intelligence engine

Where questions become answers — the core technical differentiator of the entire product
What this layer does

RAG stands for Retrieval-Augmented Generation — it means the AI doesn't answer from memory, it answers by first finding the right passages from your documents and then generating a response based only on what it found. This is what makes answers trustworthy and citable, and what separates this product from general-purpose AI tools like ChatGPT.

Why RAG matters — the exam analogy

Imagine two types of student. The first answers exam questions from memory — they might be brilliant but they can also be confidently wrong (hallucination). The second is allowed to bring a curated set of textbooks and must cite their source for every answer. The second student's answers are always verifiable. RAG makes our AI the second student, but with the speed of the first.

🔍Retriever
Given a user's question, the Retriever finds the most relevant passages across all indexed documents. It works using mathematical similarity — questions and document passages are converted to numbers, and the passages whose numbers are closest to the question's numbers are returned.
Analogy: A legal researcher who, given a question, knows which filing cabinets to open and which drawers to check — pulling the 5 most relevant pages out of 50,000 in under a second.
🏆Reranker
A second pass of quality control. After the Retriever returns 8 candidate passages, the Reranker reads each one more carefully and re-orders them by true relevance. Like a senior editor reviewing a junior researcher's notes and prioritizing the best sources.
Business impact: Improves answer accuracy significantly. The difference between returning the 3rd-best passage and the best passage can mean the difference between a vague answer and a precise one.
📝Prompt Builder
Assembles everything the AI needs: the user's question, the relevant passages found by the Retriever, and strict instructions about how to answer (cite every claim, only use provided sources, be concise). The quality of this instruction set directly determines answer quality.
✨Claude (LLM)
Anthropic's Claude model reads the assembled context and writes the answer. Claude is specifically instructed to only use the provided source passages — never to draw on its general training data. This is what prevents hallucination in this system.
Why Claude specifically: Claude has exceptional instruction-following, tends to produce well-structured answers, and is less likely to "make things up" than some alternatives when given strict sourcing requirements.
📐Embedding Model
Converts text — both documents during indexing and questions at query time — into mathematical vectors (long lists of numbers) that represent meaning. Two passages about the same topic will have similar numbers even if they use different words. This is what makes semantic search work.
👍Feedback Loop
Records when users give thumbs up or thumbs down to answers, how long they spend reading a response, and whether they ask follow-up questions. Over time, this data is used to tune the system so it gets better the longer a customer uses it — creating a switching cost advantage.
The key differentiator — what makes this different from ChatGPT

ChatGPT answers from its general training data — it can hallucinate, it can be outdated, and it cites nothing. This system only answers from the documents you've connected. Every claim has a citation. Every answer is verifiable. For enterprise use cases — legal, compliance, government contracting — that's the difference between usable and unusable.

Layer 5 — Data Layer

The memory of the system

Six specialized storage systems, each holding a different type of information the product needs to function
What this layer does

Every piece of information the system needs to store lives here — from user accounts and billing records, to the mathematical representations of 50,000 documents, to background job queues. Different types of data need different types of storage, each optimized for its purpose.

🏛️Postgres Database
The main relational database. Stores structured data: organizations, users, workspaces, document records, billing subscriptions, and the complete audit log. Think of it as the system's official record book — everything factual and structured lives here.
🧮Vector Database (pgvector)
A specialized database for storing and searching mathematical representations of text (embeddings). When a user asks a question, this database finds the mathematically closest document passages in milliseconds across millions of stored chunks. Without this, semantic search is impossible.
Analogy: A library's card catalogue, but instead of alphabetical filing, every card is filed by meaning — cards about similar topics are stored physically near each other.
📁Object Storage (S3)
Stores the actual raw files — the PDFs, Word documents, and other files that have been uploaded or synced from connectors. Cheap, infinitely scalable storage. The vector database holds the searchable version; S3 holds the original for reference and re-processing.
⚡Redis Cache
Ultra-fast temporary storage for things the system needs constantly: login sessions, rate limit counters, frequently accessed plan information, and recently computed embeddings. Prevents the main database from being asked the same question thousands of times per hour.
Analogy: A sticky note on your desk for information you need every 5 minutes, rather than going back to the filing cabinet every time.
📬Job Queue (BullMQ)
A queue of background tasks that need to happen but don't need to block the user: re-indexing a document, syncing a connector, sending a weekly email digest. Tasks are queued, processed in order, retried automatically if they fail, and logged when complete.
🤖Model Store
(Phase 4) Stores custom-trained versions of the reranking model, fine-tuned on each customer's specific query patterns. Customers who have been using the product for 6+ months benefit from a model that's been specifically optimized for their type of questions.

How connectors work — the librarian analogy

Imagine giving a librarian a special key to your company's filing rooms — one key for the Confluence room, one for the Google Drive room, one for Jira. The librarian visits each room every 4 hours, reads anything new or updated, makes detailed index cards, and files them in a central catalogue. When someone asks a question, the librarian searches the catalogue — not the original filing rooms — so the search is instant regardless of how many filing rooms exist.

"A policy organization has documents in Google Drive (research reports), Confluence (internal processes), and Notion (team wiki). An analyst asks: 'What position do we have on federal data privacy regulation?' The system searches across all three sources simultaneously, finds relevant passages in a Drive report from 2023, and answers with the citation. The analyst didn't need to know where to look — the system knew."

Questions stakeholders typically ask

Is our data safe? Could the AI leak our confidential documents?
No. The system is built with this as the primary constraint. Each organization's data is completely isolated from others (your documents are never mixed with another company's). The permission system (Layer 2) ensures each user only gets answers from documents they're already authorized to access in the source systems. And critically — your data is never used to train the underlying AI model.
What if the AI gives a wrong answer?
Every answer includes a confidence score (High / Medium / Low) and a direct citation linking back to the source passage. Users are expected to verify Medium and Low confidence answers. Importantly, when the system can't find relevant information in the connected documents, it says so clearly — it doesn't guess or fabricate. The audit log also records every answer given, so mistakes can be traced and the knowledge gaps that caused them can be filled.
Do we need to change how we store documents?
No. The product connects to the tools you already use — Confluence, Google Drive, Notion, SharePoint, etc. Your team keeps working exactly as they do now. The system reads from those tools automatically in the background. The only requirement is that documents are text-based (the system can read PDFs, Word docs, and web pages).
How long does it take to set up?
Initial setup takes under an hour. An admin connects the first data source (e.g., Google Drive), and the system begins indexing. For a typical organization with 5,000 documents, the first full index completes in 2–4 hours. After that, syncing is continuous and automatic — any document updated in Drive is reflected in the system within 4 hours.
What happens if one of the connected services (Confluence, Drive, etc.) goes down?
The product continues to work using the last indexed version of all documents. Answers are still given — they're just based on data that was accurate as of the last successful sync. The admin dashboard shows the last sync time for each connector, so admins know how current the data is.
Could a competitor or attacker access our data?
The data layer uses enterprise-grade security: all data is encrypted at rest (AES-256) and in transit (TLS 1.3). Each organization's data is in logically isolated tables with row-level security enforced at the database layer. Access is only possible via authenticated API calls. The system is designed for SOC 2 Type II compliance, which requires annual independent security audits.
